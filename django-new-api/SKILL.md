# /django-new-api — bootstrap a Django + DRF API

Create a new API project that already follows the conventions in **django-conventions**, so the first commit starts from a correct, consistent baseline instead of a bare `django-admin startproject`. The hard, repetitive part — the skeleton and all the cross-cutting plumbing — is generated by a bundled script; this workflow gathers the inputs, runs it, and brings the project to life.

Communicate progress in Spanish (the user's preferred language). All generated code is in English.

## What gets created

The scaffold lays down ~50 files: `config/settings/{base,development,staging,production,testing}.py`, `config/{urls,wsgi,asgi}.py`, and an `apps/<app>/` package with `models/abstracts/` (UUID PK, `ActiveManager`, `BaseModel`), a custom `User` + `UserManager`, `middleware/request_context.py`, `authentication/api_key.py` (dual JWT + API key), `exceptions/` (domain exceptions + global handler), `pagination/` (custom paginator), `renderers/` (standard response envelope), `admin/abstracts/`, `views/abstracts/BaseOwnedViewSet`, `serializers/abstracts/`, `tests/` (conftest + UserFactory), plus `pyproject.toml` (deps + ruff + pytest config), `.pre-commit-config.yaml`, `.python-version`, `.gitignore`, `.env.example`.

## Workflow

### 1. Gather inputs

Ask the user (one round, sensible defaults):

- **Project name** — defaults to the target directory name. Used in `pyproject.toml` and the OpenAPI title.
- **Target directory** — where to create it (must be empty or new).
- **Domain app name** — defaults to `api`. This is the first/main app (`apps.<app>`).
- **Python version** — defaults to `3.13`.

Do not ask about the cross-cutting pieces (UUID, soft-delete, auth, pagination, handler) — those are non-negotiable baseline and always included.

### 2. Run the scaffold script

```bash
python ~/.claude/skills/django-new-api/scripts/scaffold.py \
  --target <target-dir> \
  --app <app-name> \
  --name <project-name> \
  --python <x.y>
```

The script never overwrites existing files unless `--force` is passed, so it is safe to re-run. It only writes files — it does not touch uv, the database, or git.

### 3. Bring the environment up with uv

From the target directory:

```bash
cd <target-dir>
uv python pin <x.y>      # if not already pinned by the scaffold
uv sync                  # create .venv, resolve, write uv.lock
uv run pre-commit install
git init                 # if this is a fresh directory
```

If `uv` is unavailable, tell the user and stop — this baseline assumes uv (see **python-tooling**). Do not silently fall back to pip.

### 4. Database, migrations, superuser

The settings expect PostgreSQL (configured via env vars / `.env`). Make sure a database is reachable (the user may run it via Docker or locally), then:

```bash
cp .env.example .env     # then fill SECRET_KEY, API_KEY, DB_* values
uv run python manage.py makemigrations <app>
# inspect apps/<app>/migrations/0001_initial.py before applying
uv run python manage.py migrate
uv run python manage.py createsuperuser   # optional, for /admin
```

Generating migrations only via `makemigrations` and inspecting them before `migrate` is part of the conventions — never hand-write the initial migration.

### 5. Verify

Confirm the project boots and the suite is green before handing off:

```bash
uv run python manage.py check
uv run ruff check . && uv run ruff format --check .
uv run pytest
```

`manage.py check` must report no issues. Report the result to the user with the project path and how to run it (`uv run python manage.py runserver`), plus the docs URL (`/v1/docs/`).

### 6. Point to the next step

The scaffold creates the plumbing but **no domain endpoints** — `apps/<app>/urls.py` has an empty router. To add the first resource (model + serializer + viewset + admin + tests), use the **django-feature** skill. The layer references — **django-orm-patterns**, **django-rest-framework**, **django-cbv-patterns**, **django-testing** — describe the exact patterns; **django-conventions** is the rulebook they all serve.

## Notes

- The script's defaults (dependency versions, ruff rules, JWT lifetimes) mirror the baseline; adjust `pyproject.toml` afterwards if the user needs different pins.
- The `API_KEY` in `.env` gates the API-key auth path for public endpoints. Generate a strong value; never commit the real `.env`.
- If the user wants Docker (Postgres + the API), that is project-specific infrastructure — set it up separately and document it in the project's own `docs/` (see the project-docs skill), not in this baseline.
